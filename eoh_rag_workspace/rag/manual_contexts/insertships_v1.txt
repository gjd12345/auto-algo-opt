Section 1 - Algorithm flow / 算法流程
Goal: produce one safe InsertShips heuristic, not a simulator shortcut. The function should iterate each dynamic order i from 0 to total_ship-1 and treat oris[i], dess[i] as a pair. First try feasible insertion into existing assignments, then fall back to a simple seed-style AddShip path when no improvement is found. Pseudocode:
- for i in orders: bestAssign = -1; bestCost = +Inf
- for each assign index a < dispatch.AssignsLen: clone or temporarily add; call AddShip(i, ori, des); if ok, GenRoute, estimate Cost or TotalCost delta; undo if temporary.
- choose the candidate with lowest incremental cost, not the globally smallest fake number.
- if bestAssign >= 0: apply the selected insertion and regenerate route.
- else: try any existing assignment that accepts AddShip; if all fail and capacity allows, initialize/use a new assignment consistently with the project structs.
- after all orders are processed, call dispatch.RenewnTotalCost() exactly once immediately before return.
中文提示: 优先保持订单不丢失, 先可行再优化. 不要因为某一单插入失败就 break 外层循环; continue 到下一单只应发生在已经完成安全处理之后. Keep the code local, deterministic, and compatible with the existing Go file.

Section 2 - Safety constraints / 安全约束
The prompt material is reference only. Generated code must obey the output contract: return only func InsertShips(...), no package line, no imports, no markdown, no explanations. Do not print logs, sleep, read files, use randomness that changes reproducibility, spawn processes, or depend on network/time/environment. Do not overwrite dispatch.TotalCost or AccumulatedCost with magic constants; the evaluator expects RenewnTotalCost to compute the final cost. Do not mock cal_dis, Dispatch, Assign, Station, or simulation helpers. Avoid unsafe assumptions about array length: respect MAXASSIGNS, AssignsLen, len(oris), len(dess), and total_ship. Bounds logic should use n := total_ship then clamp n to len(oris) and len(dess) if needed. 中文安全线: 不要为了拿低分而跳过订单、直接返回空 dispatch、删除已有 Assign、或把费用设成 0. If temporary mutation is used for scoring, restore the exact previous state before trying the next candidate, otherwise later candidates inherit corrupted routes. Prefer simple loops and explicit checks over clever abstractions. Always leave dispatch internally consistent: AssignsLen non-negative, no access beyond Assigns, and routes regenerated after successful AddShip.

Section 3 - Failure experience / 失败经验
Observed bad candidates often looked good by objective but failed real validation. Common patterns: early break when bestIndex == -1, which silently drops remaining orders; setting TotalCost to a small number without changing routes; returning before RenewnTotalCost; optimizing only the first request; using imports or helper functions not present in the target file; and mutating an assignment during evaluation without undo. A robust heuristic can be boring: nearest-current-station first, then lowest incremental distance, then fallback. Example scoring idea: score = cal_dis(assign.StationCurrent, ori) + cal_dis(ori, des); add a small penalty for already long or busy assignments if Cost is available. Tie-break deterministically by lower score, then lower assignment index. 中文经验: 低密度场景里简单策略往往够用, 高密度时更需要保证每单都有兜底插入. If AddShip returns false, try another assignment rather than abandoning the order. If new assignment creation is uncertain because struct fields are not fully known, safer fallback is to reuse an existing feasible assignment. The final code should be short enough for the LLM to maintain and should prioritize correctness over exotic local search.
